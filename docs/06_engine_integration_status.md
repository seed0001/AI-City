# Document 6 — Engine Integration Status (Truth Report)

This document is the honest answer to "are the engines actually running the residents?" It is not a marketing document. It tells you exactly which engines are wired into observable behavior, which are wired in name only, and which are deliberately out.

The numbers below were generated by a real run of `EngineBundle.create + tick + synthesizeDecision + synthesizeConversationContext + recordEvent`. They reflect the current state of the integration as of the writing of this document.

---

## 1. Headline numbers

- Total importable classes across `Engines.{emotion,personality,memory,cognitive,behavior,utility}`: **119**
- True engine classes (after dataclass / enum / value-record filter): **103**
- Excluded data containers (deliberately not instantiated): **16**
- Engines instantiated per resident at full bundle creation: **103**
- Composite engines wired with sibling injection (advanced_pairing): **4**
- Engines disabled at runtime due to instantiation failure: **0**
- Typed adapters available (engines with explicit, type-safe wrappers): **16**
- Engines with at least one capability-mapped method (decision / state / event / expression): **~80**
- Engines whose output is captured per tick (`contributingEngines`): **~73 of 103** measured in a two-resident smoke run.

The 103-of-103 active count is real. Every compatible engine class instantiates cleanly per resident with its own state directory. The 0-of-103 disabled count is also real. There are no silent failures in the inventory.

The number that used to matter most was the typed-adapter count of **16 of 103**. As of the engine influence expansion (Phase 1–7 in Doc 10) that framing is incomplete. The 16 typed adapters remain the engines with hand-written, signature-correct wrappers; they are still the engines whose outputs land most reliably. But the bundle now also runs a per-engine `GenericEngineAdapter` that uses a one-time capability scan plus signature inspection to safely call the right method per phase on every other engine, capture its output, and feed it into either decision synthesis (when the output is intent-shaped) or the conversation `extendedContext` block (when the output is state-shaped). In a smoke run, **~73 of 103 engines contribute captured output** and **the conversation context surfaces ranked summaries from up to 8 engines** outside the 7 named core fields. The "16 active brains + 87 passengers" framing is no longer accurate. The accurate framing is "16 hand-written typed adapters + ~57 capability-driven generic contributors + ~30 silent engines."

## 4. Status categories applied to the inventory

Below, every active engine is categorized by what it actually does in the running system today. The categories are honest, and they have shifted since the engine influence expansion landed.

- **TYPED ADAPTER** — engine is invoked through a hand-written adapter in `engine_adapters.py`. Outputs feed decision synthesis, conversation context, and event handling reliably and with the correct argument shapes.
- **CAPABILITY DECISION CONTRIBUTOR** — engine has a decision-shaped method (`select_behavior`, `decide`, `recommend_action`, etc.) that the `GenericEngineAdapter` invokes via signature-aware dispatch. Its votes enter `synthesizeDecision` weighted by role priority.
- **CAPABILITY CONTEXT CONTRIBUTOR** — engine has a state-shaped method whose output is summarized into the conversation `extendedContext` block, ranked by `CONTEXT_PRIORITY`. Up to 8 such summaries surface per conversation request.
- **CAPABILITY EVENT ABSORBER** — engine has an event-shaped method that fires on every `recordEvent`. The `GenericEngineAdapter` builds the right argument shape (dict / str / primitive) from the event payload.
- **PASSIVE** — engine ticks through phases but does not contribute observable output. Usually because its capability scan returned no methods or because its outputs are intentionally diagnostic-only.
- **DISABLED** — engine fails at runtime (currently zero).

An engine can occupy multiple categories — most of the typed-adapter engines also appear in the capability buckets because the registry scans them too (the typed adapter still wins at call time, but the capability summary appears in `debug_snapshot.capabilities` for visibility).

### 4.1 TYPED ADAPTER (16 engines, hand-written)

These are listed in Section 3 above. Their outputs measurably shape behavior with full type-correctness.

### 4.2 CAPABILITY DECISION CONTRIBUTOR (~10 engines beyond typed)

The bundle's capability registry currently maps decision-shaped methods on roughly **10 engines** beyond the typed set, most of them `CONTROL` or `COGNITION`:

- `BehavioralPulseEngine`, `InitiativeSchedulerSeed`, `LifeSchedulerLyra`, `CycleEngineGramps`, `SeedDynamics`, `DailyRhythmLyra` — schedulers and rhythm engines whose `select_behavior` / `decide` / `get_current_phase` methods are now picked up cleanly.
- `ActionEngineGramps`, `DeliberationEngineGramps`, `PreThinkEngineConnor` — cognition engines with `decide` / `recommend_action` / `next_action` surfaces.
- A handful of others surface `get_top_intent`-like outputs.

Each of these contributes a vote to `synthesizeDecision` weighted by `DECISION_PRIORITY[role] * source_bonus[method]`. Drive-shaped methods get the heaviest source bonus (1.4×), then goal-shaped (1.3×), intent-shaped (1.2×), generic decision (1.0×), phase (0.7×). Three additional bias channels also vote:

- `bias.emotional_pressure` — reads cached emotion-engine state and votes `go_home` on high anxiety, `seek_social` on high joy, etc.
- `bias.personality_consistency` — reads cached personality traits and votes `seek_social` for extraversion, `pursue_daily_objective` for conscientiousness, `wander` for curiosity.
- `bias.world_state_pressure` — converts the snapshot fields the world handed in (hunger, energy, socialTolerance, mood) into explicit votes. This is the engine layer reading body-state. Without it, two freshly-initialized residents converge to the same default-engine intent on first tick.

Every contributor — typed engine, capability engine, or bias channel — appears in the response's `contributingEngines` list and in `last_decision_breakdown`, with role / method / weight / source bonus visible to the HUD.

### 4.3 CAPABILITY CONTEXT CONTRIBUTOR (~74 engines)

Roughly **74 of the 103 engines** have at least one state-shaped method (`get_state`, `get_summary`, `get_traits`, `get_active_goals`, etc.). The bundle's `_build_extended_context` pulls cached output from each, summarizes it (`summarize_engine_output` produces ≤120-char prompt-safe strings sorted by absolute magnitude for numeric dicts), ranks by `CONTEXT_PRIORITY` (`EMOTION > MEMORY > PERSONALITY > COGNITION > CONTROL > EXPRESSION > UTILITY`), and caps at **8 per conversation context** with per-role caps to prevent any one role flooding (e.g. cognitive at 4, memory at 3, personality at 3, emotion at 3, expression / control / utility at 2).

In a typical smoke run, the 8 `extendedContext` entries span all four major cognition roles. Examples:

- `mental_health_beast.MentalHealthEngineBeast` — `depression=0.00, mania=0.00, ptsd=0.00, withdrawal=0.00`
- `memory_reconciliation_gramps.MemoryReconciliationEngineGramps` — `reconciliation_active=0.00, onboarding_active=0.00, …`
- `personality_seed.PersonalitySeed` — `patience=0.70, curiosity_level=0.50, verbosity=0.50, initiative=0.40`
- `personality_lyra.PersonalityEngineLyra` — `playful=0.90, calm=0.50, focused=0.50, tired=0.50`
- `cognitive.advanced_pairing_engine.advanced_pairing_engine` — `pairs=, stats=total_pairs=0.00, total_quer, history=, patterns=`

This list is parallel to the 7 core `engineBrainContext` fields (it does not replace them). The 7 core fields remain the LLM prompt spine; `extendedContext` is available to the HUD and to richer prompt experiments.

The 7 core fields and the engines that fill them:

- `emotionalState` — `EmotionKernelGramps` (or peer)
- `relationshipReasoning` — `RelationalMemorySystem`
- `currentIntent` — `IntentSystem`
- `activeGoals` — `GoalEngineGramps`
- `driveState` — `DriveModelSeed`
- `selfNarrative` — `SelfModelGramps`
- `recentEpisodes` — `EpisodeMemoryGramps`

These 7 are now joined by the **`extendedContext`** block in every response, so the conversation context *as a whole* now reflects the full breadth of the engine state, not just the 7 core engines.

### 4.4 CAPABILITY EVENT ABSORBER (~19 engines beyond typed)

Roughly **19 engines** have event-shaped methods (`record_event`, `process_event`, `ingest_event`, `update`, `score_interaction`, `process_input`, `receive_input`, `append_to_narrative`, `add_intent`, `add_word`, etc.) that the `GenericEngineAdapter.absorb_event` invokes on every `recordEvent`. The adapter is signature-aware — it passes the event dict, `event.summary`, `event.tone`, partner ids, or a sensible default per parameter, swallowing per-call exceptions so one engine's mismatch never breaks propagation.

Every event is also tagged via `classify_event_tags(event)` before propagation. Tags are `["emotional"|"social"|"goal-related"|"memory-worthy"|"conflict"|"positive"|<eventType>|"ambient"]`. Tags appear on the bundle's `last_event_tags` field and in the debug HUD. They are informational right now (engines do not gate on them yet); they exist to make the propagation breadth visible.

### 4.5 PASSIVE

- `HallucinationMonitorGramps`
- `DashboardEngineBeast`
- `LambdaPsiInstrumentationConnor`
- `ObserverClientSeed`

These four are explicitly classified `PASSIVE_MONITOR`. They tick along, expose state for diagnostics, and never affect decisions or dialogue. The capability scan still inspects them, but they are excluded from `extendedContext` ranking.

### 4.6 DISABLED

Zero engines are currently disabled at runtime. Every compatible class instantiates cleanly. The `disabled_engines` dictionary is empty in healthy operation. The `GenericEngineAdapter` swallows per-call exceptions rather than propagating them, so a method that fails on one tick does not retire its engine; it simply produces no output for that tick.

## 7. What is actually driving NPC behavior right now

Honest answer, three layers deep, after the engine influence expansion landed:

### 7.1 Decisions

When a resident decides what to do next:

- The world layer asks the brain via `/brains/decision`. The brain runs the full bundle, then aggregates votes via weighted voting in `synthesizeDecision`.
- The vote ledger collects signals from four sources, in this order:
  1. **Typed-adapter engines** with class-aware intent extraction (`DriveModelSeed → get_active_drive`, `IntentSystem → get_top_intent`, `GoalEngineGramps → get_active_goals`, `DailyRhythmLyra → get_current_phase`).
  2. **Capability decision contributors** — every other engine with a decision-shaped method, called via the `GenericEngineAdapter` and run through the strict `_normalize_intent` filter (refuses to chase substrings inside `str(some_dict)` to avoid false votes).
  3. **Inferred biases** — `bias.emotional_pressure` (reads cached emotion-engine state), `bias.personality_consistency` (reads cached personality traits), `bias.world_state_pressure` (reads the snapshot the world handed in: hunger, energy, socialTolerance, mood).
  4. **Vote weighting** — `weight = DECISION_PRIORITY[role] * source_bonus[method] * confidence`. Drive surfaces (1.4×) and goal surfaces (1.3×) are biased loudest, per the user's specification.
- In a smoke run with two contrasting residents (Ada: extraverted, hungry, mood=calm; Bex: reserved, tired, mood=nervous), decisions diverged on first tick: Ada → `seek_social` (drive model + world-state seek_social vote), Bex → `idle` (schedulers + action engine). Two residents with default-init engines but different snapshot context now produce visibly different intents.
- The response includes `contributors` (engine keys whose vote landed on the winning intent), `contributingEngines` (full per-engine breakdown with role / method / weight / source bonus), and `source` (`"full_brain_synthesis"` or `"fallback"`).
- If the brain produces an intent with confidence ≥ 0.25, the world follows it. Otherwise, the world's heuristic fallback runs.

### 7.2 Conversation context

When a resident speaks:

- The world layer awaits engine context for both speakers via `/brains/conversation-context`. This call is required when the brain is online; the conversation system does not skip it.
- The bundle returns:
  - The 7 core `engineBrainContext` fields (unchanged — this remains the LLM prompt spine: `emotionalState`, `relationshipReasoning`, `currentIntent`, `activeGoals`, `driveState`, `selfNarrative`, `recentEpisodes`).
  - **`extendedContext`** — up to 8 ranked engine summaries from the wider engine state, each ≤120 chars, ranked by `CONTEXT_PRIORITY` with per-role caps. This is parallel to the 7 core fields, not appended into them. It is available to the HUD and to richer prompt experiments. The default LLM prompt still uses the 7 core fields by design (token budget protection).
  - **`contextSources`** — full list of which engine fed which field (`{ engineKey, field, role }`), 7 CORE entries plus one per extendedContext signal. This makes the source of every prompt-shaping signal traceable.
- The conversation prompt places the 7 core fields at the top of the user message. The LLM (when present) is instructed to make every spoken line visibly reflect them. Without an LLM, the engine-aware stub weaves the same fields into its lines.
- The `conversationSource` flag reports `"engine"` when both speakers had engine context this tick; `"fallback"` otherwise.

### 7.3 Event feedback

When a conversation finishes:

- The world layer sends a structured `conversation_outcome` event to each speaker's bundle.
- Before propagation, the event is tagged via `classify_event_tags(event)`. A typical conversation outcome surfaces tags like `["conversation_outcome", "social", "emotional", "memory-worthy"]`, with a tense outcome adding `"conflict"` and a warm outcome adding `"positive"`. Tags appear on `last_event_tags` and in the debug HUD.
- Six typed-adapter engines respond with full type-correctness: `RelationalMemorySystem.store_memory`, `IntentSystem.add_intent`, `DriveModelSeed.update_from_interaction`, `GoalEngineGramps.update_goal_progress`, `EmotionKernelGramps.update`, `SelfModelGramps.append_to_narrative`. Secondary effects fire through `MemoryManagerLyra`, `WordStoreSeed`, `MentalHealthEngineBeast`, `EmotionalFeedbackBeast`, `ConnorStateBeast`.
- The remaining engines that have any event-shaped method (~19) are now invoked via the `GenericEngineAdapter.absorb_event`, which passes the right argument shape per signature. Per-call exceptions are swallowed; one engine's mismatch never stops propagation.

This is the closed feedback loop that lets the next conversation reflect the previous one. The smoke test in the system: a tense conversation with a partner produces a `Follow up with <partner> about tension` intent on the speaker's `IntentSystem`, which surfaces in the next `engineBrainContext.currentIntent` (verified end-to-end in the upgrade smoke).

## 8. What is NOT wired yet

Honest list, post-expansion:

1. **Most cognition engines still have no typed adapter** — but their state is now captured. ThoughtEngineGramps, InnerMonologueGramps, DeliberationEngineGramps, ActionEngineGramps, DeepReviewEngineGramps, JournalingEngineBeast, SkillAcquisitionEngineGramps, LLMDriftAnalysisEngine, RegulatorConnor, LambdaPsiEngineConnor, QuantumReasoningEngineConnor, the entire `real_intelligence_connor` set, the entire `cognitive_engine` set, the entire `neural_emergence_connor` set, `CreativeEngines`, `SelfRegulationEngineBeast` — all instantiated, all ticked. Many of them now surface in `extendedContext` because they expose `get_state` (which the capability scan picked up). What they still lack is *type-correct* event handling and *intent-shaped* decision votes. A typed adapter will still produce sharper output than the generic adapter for any of these.

2. **Personality engines other than `SelfModelGramps` are now contributing through extendedContext.** `PersonalityEngineBeast`, `PersonalityEngineLyra`, `PersonalitySeed`, and (when state is available) `IdentityEngine`, `SelfModelSeed`, `SocialCircleLyra`, `StoryOfSelfSeed` are surfaced as ranked summaries in the conversation `extendedContext` block. Their dynamics are not yet routed *into* the 7 core `engineBrainContext.selfNarrative` field — that still pulls only from `SelfModelGramps`. A typed adapter for each that contributes a structured trait + identity blob is the next step (Doc 9 §3.4).

3. **Memory engines beyond the typed three now feed extendedContext** but not the LLM prompt. `KnowledgeBaseGramps`, `MemoryCoreGramps`, `MemoryReconciliationEngineGramps`, `SeedMemory`, `TemporalContinuitySeed`, `UnifiedMemorySeed`, `VectorMemoryGramps` — their state appears in `extendedContext`. To enter the LLM prompt they would need to either be promoted into the 7 core fields or have the prompt builder opt into reading `extendedContext`. The decision to keep the prompt anchored on the 7 core fields is deliberate (token budget).

4. **Composite engines beyond advanced_pairing are not wired.** Several connor / beast engines have internal compositions that could be wired with sibling injection but are not yet. The four advanced_pairing composites are the only ones with sibling-aware wiring today.

5. **`RelationalMemorySystem.get_reasoning_summary` reads from a global `reasoning_chains.json`.** That file is in the working directory of the brain service, not in the per-resident state directory. The write-side adapter is per-resident (it writes `store_memory` correctly with partner tags), but the read-side surface is global. Two residents currently see the same reasoning summary when they query this engine. This is technical debt; Document 9 includes the fix plan.

6. **Many utility engines remain unconsumed.** `WebCrawlerGramps`, `WebLearnerSeed`, `VocabularyEnforcerSeed`, `FallbackPhrasesSeed`, `CuriosityEngineSeed`, `AgingEngine`, `AudioEngine`, `CharacterVoiceSeed`, `DJMusicEngineBeast`, `ImageGenerationEngine`, `VisionEngine`, `VisionModuleGramps`, `VoiceModulationEngineBeast`, `AvatarGeneratorLyra`, `OllamaClientConnor`, `SystemLoadBalancer`. The capability scan finds state methods on most; they may surface in extendedContext if their relevance score beats more impactful engines (rare). The world layer does not pull from any of them directly. This is mostly correct — utility engines should remain ancillary.

7. **The TownEntity world data is still not fully reflected back into engines.** The `tickContext` and `decisionContext` payloads carry `mood`, `energy`, `hunger`, `socialTolerance`, `currentGoal`, `currentAction`, `lifeAdaptation`, `townDaysLived`. The new `bias.world_state_pressure` channel reads these, so they do reach decision synthesis. They do not yet reach engines that would benefit from richer payloads — the full `dailyPlan`, the `relationships` map, the `townRoleOptions` are not being passed.

8. **`extendedContext` is populated but not yet injected into the LLM prompt.** It appears in the API response and in the HUD. The LLM prompt builder still uses only the 7 core fields. Wiring `extendedContext` into the prompt is a token-budget decision and a quality experiment that has not been run.

## 9. Honest limitations

A short list of things that are true today and worth saying plainly:

- **The 16-vs-87 framing is replaced by a three-tier framing.** Tier 1: 16 typed adapters with sharp, type-correct outputs. Tier 2: ~57 capability-driven generic contributors that surface state into `extendedContext` and event absorption, plus ~10 with decision-shaped methods that vote in `synthesizeDecision`. Tier 3: ~30 silent engines (cognitive engines whose state methods returned no captured output, or engines whose outputs are too cryptic to summarize cleanly).
- **The decision pipeline now diverges between residents on the first tick.** Two residents with different snapshots produce different intents without waiting for engine state to drift. That was a real success criterion; it now passes.
- **The conversation context is dramatically richer.** A fresh resident with no event history still has empty or near-empty `recentEpisodes` and `relationshipReasoning` (those only fill from event history). But `extendedContext` is non-empty from tick zero, with ranked summaries from emotion / memory / personality / cognition engines. The HUD shows this immediately.
- **Per-resident isolation is real for engines that respect their `state_path`.** A small number of engines write to fixed file names — those still need to be wrapped or patched per-resident.
- **The strongest honest statement is now:** AI City has a real, working engine-driven cognition layer with three coordinated tiers — typed adapters for the load-bearing engines, a capability-driven generic adapter for the rest, and three explicit bias channels that read body state, emotional pressure, and personality traits. The remaining work is mostly about turning more Tier-2 engines into Tier-1 typed adapters where their domain demands it (Doc 9 §3.1), and deciding whether to inject `extendedContext` into the LLM prompt.

Documents 7 and 9 describe what that means at runtime and what would need to change to advance it. Document 10 summarizes the seven-phase upgrade that landed the capability registry, generic adapter, weighted voting, extendedContext, event tagging, priority tables, and debug visibility.
